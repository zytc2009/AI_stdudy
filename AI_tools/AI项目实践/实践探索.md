# Claude Code 团队实施方案

[toc]

> 适用团队：C++ 跨平台 SDK + 平台层，共 30 人
> 技术栈：C++ Core / JNI / ObjC / Swift / Kotlin
> 代码管理：内网 GitLab
> 开发环境：Mac + Windows (WSL2)

---

## 一、总体原则

- **分批推进**，不全员同时铺开
- **从最痛的点开始**（跨平台胶水代码、单元测试补全）
- **工具配置统一管理**，通过 Git 仓库分发
- **数据不出内网**，API 流量走内网代理

## 二、基础设施准备

### 1. API 接入方式（三选一）

| 方案                     | 安全性               | 成本 | 实施难度 | 推荐场景         |
| ------------------------ | -------------------- | ---- | -------- | ---------------- |
| 直连 Anthropic API       | 中（数据出境）       | 低   | 简单     | 试点阶段临时使用 |
| AWS Bedrock / GCP Vertex | 高（数据留在云账号） | 中   | 中       | **推荐长期方案** |
| 自建内网代理             | 最高（流量可审计）   | 高   | 复杂     | 合规要求严格时   |

**推荐路径**：试点期直连 → 正式推广切换 Bedrock/代理

### 2. 内网代理搭建（正式推广前完成）

```bash
# 用 LiteLLM 搭建轻量代理（一台内网 Linux 服务器）
pip install litellm[proxy]
litellm --model bedrock/anthropic.claude-sonnet-4-6 \
        --port 8080
```

代理的价值：

- 统一管理 API Key，不用每人持有
- 记录各团队 token 用量（成本分摊）
- 一处更换 Key，全团队生效

### 3. 配置仓库（建立统一管理）

## 四、各仓库 CLAUDE.md 模板

### SDK Core

```markdown
# SDK Core - Claude Code Guidelines

## 项目背景
跨平台 C++ SDK，支持 Windows / Android / iOS / Mac

## 禁止操作
- 不得修改 include/ 下的公共头文件（破坏 ABI）
- 不得删除带 DEPRECATED 注释的函数
- 不得直接 push 到 main/release 分支

## 代码规范
- C++17 标准，禁用 C++20 特性（部分平台编译器不支持）
- 内存管理用 RAII / smart pointer，禁止裸 new/delete
- 跨平台条件编译统一用 platform_defs.h 中的宏

## 跨平台接口一致性
新增或修改接口时，必须同时检查三端对应声明：
- C++ 核心: include/sdk/
- Android:  android/src/main/kotlin/sdk/
- iOS/Mac:  apple/Sources/SDK/

## 构建方式
cmake -B build -DCMAKE_BUILD_TYPE=Debug -DBUILD_TESTS=ON
cmake --build build
ctest --test-dir build --output-on-failure

## 单元测试规范
新增功能必须有对应 GTest 用例，包含：
1. 正常路径
2. 边界条件（空指针、空字符串、最大/最小值）
3. 错误路径（无效参数、资源不足）
4. 平台差异（每个 #ifdef 分支都要覆盖）

## ObjC → Swift 迁移规范
- ObjC block callback → Swift async throws 或 AsyncStream
- NSString/NSArray → String/Array（Swift 原生类型）
- 旧 ObjC 接口保留，加 @available(*, deprecated) 标注
- 不允许 Claude 自动删除旧 ObjC 接口

## JNI → Kotlin 规范
- JNI external fun 集中在 NativeBridge.kt
- 上层 API 用 Kotlin 封装，不直接暴露 external fun
- 回调用 Kotlin Flow 或 suspend fun 包装
```

### 平台层

```markdown
# Platform Layer - Claude Code Guidelines

## 依赖边界
- SDK 调用只通过 sdk/include/ 暴露的接口
- 不得绕过 SDK 直接调用系统 API
- 不得修改 sdk/ 子模块内容

## 分支策略
feature/* → develop → main
hotfix/* → main → 同步 develop

## 禁止
- 访问 ../product/ 目录
- 修改 SDK 子模块内容
```

### 测试团队

```markdown
# QA - Claude Code Guidelines

## 权限范围
- 只读访问 SDK 和平台层代码
- 可运行测试，不可修改业务代码

## 主要用途
- 将黑盒测试场景描述转化为 GTest 用例
- 生成自动化回归测试脚本
- 分析测试覆盖率报告
```

## 五、单元测试补全策略

### 覆盖率现状分析

```bash
cmake -B build -DCMAKE_BUILD_TYPE=Debug \
      -DCMAKE_CXX_FLAGS="--coverage" \
      -DBUILD_TESTS=ON
cmake --build build
ctest --test-dir build
lcov --capture --directory build --output-file coverage.info
lcov --remove coverage.info '*/tests/*' '*/third_party/*' \
     --output-file coverage_filtered.info
genhtml coverage_filtered.info --output-directory coverage_html
```

### 补测试优先级

```
高优先级（立即补）:
├── 公共 API 层 include/ 下的所有接口
├── 平台差异代码（#ifdef WIN32 / APPLE / ANDROID 分支）
└── 所有错误处理路径

低优先级（暂缓）:
├── 内部工具函数（有上层测试间接覆盖）
└── 第三方库封装层
```

### 给 Claude 的标准 Prompt

```
读取 [文件路径]，为以下函数生成 GTest 用例：
[函数名]
要求：
- 正常路径
- 所有参数的边界值
- 明显的错误路径
- 与 tests/ 目录现有风格保持一致
- 不依赖真实硬件设备和网络连接
```

### GMock 隔离硬件依赖（阶段二）

```cpp
// 定义硬件抽象接口
class IAudioDevice {
public:
    virtual ~IAudioDevice() = default;
    virtual ErrorCode open(const std::string& deviceId, int sampleRate) = 0;
    virtual void close() = 0;
    virtual ErrorCode read(uint8_t* buffer, size_t size) = 0;
};

// Mock 实现（自动化测试用）
class MockAudioDevice : public IAudioDevice {
public:
    MOCK_METHOD(ErrorCode, open, (const std::string&, int), (override));
    MOCK_METHOD(void, close, (), (override));
    MOCK_METHOD(ErrorCode, read, (uint8_t*, size_t), (override));
};
```

## 六、GitLab CI 配置

### 基础版（立即可用）

```yaml
# .gitlab-ci.yml
stages:
  - build
  - test

build:linux:
  stage: build
  image: ubuntu:22.04
  before_script:
    - apt-get update -q
    - apt-get install -y cmake ninja-build g++ libgtest-dev
  script:
    - cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=Debug -DBUILD_TESTS=ON
    - cmake --build build
  artifacts:
    paths:
      - build/
    expire_in: 1 hour

test:linux:
  stage: test
  image: ubuntu:22.04
  needs: [build:linux]
  script:
    - ctest --test-dir build --output-on-failure --parallel 4
  artifacts:
    when: always
    reports:
      junit: build/test_results.xml
```

### 覆盖率报告（第二阶段加入）

```yaml
test:coverage:
  stage: test
  image: ubuntu:22.04
  script:
    - apt-get install -y lcov
    - cmake -B build -DBUILD_TESTS=ON
        -DCMAKE_CXX_FLAGS="--coverage"
        -DCMAKE_EXE_LINKER_FLAGS="--coverage"
    - cmake --build build
    - ctest --test-dir build
    - lcov --capture --directory build --output-file coverage.info
    - lcov --remove coverage.info '*/tests/*' '*/third_party/*'
        --output-file coverage_filtered.info
    - genhtml coverage_filtered.info --output-directory coverage_html
  coverage: '/lines\.\.\.\.\.\.: (\d+\.\d+)%/'
  artifacts:
    paths:
      - coverage_html/
```

### 多平台矩阵（第三阶段加入）

```yaml
.build_template: &build_template
  script:
    - cmake -B build -DBUILD_TESTS=ON -DCMAKE_BUILD_TYPE=Debug
    - cmake --build build
    - ctest --test-dir build --output-on-failure

build_test:linux:
  <<: *build_template
  image: ubuntu:22.04
  tags: [linux]

build_test:macos:
  <<: *build_template
  tags: [macos-runner]

build_test:windows:
  tags: [windows-runner]
  script:
    - cmake -B build -DBUILD_TESTS=ON
    - cmake --build build --config Debug
    - ctest --test-dir build -C Debug --output-on-failure
```

## 十、常见阻力应对

| 阻力                 | 应对方式                                              |
| -------------------- | ----------------------------------------------------- |
| "AI 写的代码不可信"  | 规定 Claude 生成的代码必须人工 review，与普通代码一样 |
| "学习成本高"         | 第一批成员输出内部教程，降低后续门槛                  |
| "我的场景 AI 不适合" | 先从最适合场景（写测试）开始，不强推所有场景          |
| "安全顾虑"           | 内网代理 + API Key 统一管理，数据不出内网             |
| "影响编码习惯"       | 强调 Claude 是辅助工具，最终决策权在开发者            |
